# Developer Journal

## 1. Holistic System Functionality
The "Eligibill" application was engineered to act as a complete, end-to-end Clinical Decision Support System. Holistically, the code functions as a three-stage pipeline designed to automate oncology cohort discovery:
* **Stage 1: Ingestion & Security:** The application initialises a secure `customtkinter` frontend, guarded by a mock NHS domain authentication gate. Once authenticated, the UI uses `tkinterdnd2` to allow users to drag and drop large, unstructured JSON datasets directly into the application's memory.
* **Stage 2: Deterministic Extraction (NLP):** The backend engine parses the text line-by-line using highly specific Regular Expressions to hunt for and extract core oncology metrics: Age, Prostate Volume, Gleason Score, PSA Levels, and Clinical T-Stage.
* **Stage 3: Business Logic & Export:** These extracted values are fed into an Object-Oriented Rule Engine. The engine evaluates the metrics against NICE NG131 guidelines using a "Cascading OR" logic structure. Finally, it dynamically writes the processed cohort into a structured `.csv` audit trail. To ensure the UI does not freeze during this batch processing, the execution loop runs on a secondary daemon thread.

## 2. The Structural Logic of Object-Oriented Programming (OOP)
To ensure the codebase remained scalable, maintainable, and clinically safe, I implemented a strict parent-child inheritance structure for the data models, drawing heavily on Lecture 9:
* **The Superclass (`PatientRecord`):** This base class encapsulates universal medical attributes that any human patient would possess (e.g., `patient_id`, `age`, and default routing statuses).
* **The Subclass (`UrologyPatient`):** Inheriting from `PatientRecord`, this class utilises the `super().__init__()` function to pull in the parent attributes while introducing specialised, domain-specific properties required for oncology (e.g., `gleason`, `psa`, `t_stage`).
* **The Rationale:** This architecture is vital for system integrity. By isolating the `evaluate_eligibility()` logic entirely within the `UrologyPatient` subclass, I ensured that highly specific business rules are never accidentally applied to generic patient records. It also significantly extends the code's extensibility; new subclasses can be easily added for different disease states without rewriting the universal foundation.

## 3. A Coding Struggle and Its Resolution
The most significant technical hurdle I encountered involved integrating modern user interface components with native OS-level file mechanics. I wanted to upgrade the visual aesthetics using `customtkinter` while simultaneously allowing drag-and-drop JSON file ingestion via `tkinterdnd2`.
* **The Problem:** Because both of these third-party libraries attempt to act as the "root" application window and initialise underlying Tcl/Tk configurations, running them together caused immediate class conflicts and application crashes.
* **The Solution:** I solved this by utilising multiple inheritance and dynamic argument packing (`*args` and `**kwargs`). I engineered a custom wrapper class (`TkinterDnD_CTk`) that inherited from both libraries simultaneously. By passing arguments through the `__init__` constructor, I allowed Python to dynamically funnel the visual styling parameters up to the `CustomTkinter` parent class, while simultaneously calling `TkinterDnD._require(self)` to hook the drag-and-drop extension into the underlying system. This bridged the libraries flawlessly.

## 4. Explicit Connections to Course Materials
The engineering of this project relied heavily on the foundational concepts taught throughout this module:
* **Lecture 3 (Logic & Flow Control):** The `evaluate_eligibility()` function relies heavily on chained conditionals (`if` / `elif`) to apply the NICE clinical guidelines. By organising the checks from highest risk to lowest, I utilised implicit logic—meaning if the code reaches a certain `elif` statement, it intrinsically knows the prior conditions were False.
* **Lecture 6 (Exception Handling & Dictionaries):** I implemented `try...except` blocks within the `app.py` thread execution loops. This gracefully handles data parsing errors without crashing the frontend UI. Additionally, I heavily relied on lists of dictionaries to temporarily store processed patient states before export.
* **Lecture 7 (Libraries):** To elevate the project, I implemented external GUI libraries alongside standard built-in modules (`json`, `csv`, `threading`, `re`), directly applying the principle of importing and leveraging pre-built functionality ("standing on the shoulders of giants") to add advanced system features.
* **Lecture 8 (Unit Testing & File I/O):** Inspired by the `assert` testing practices, I utilised Test-Driven Development via `unittest` in `test_engine.py` to verify my Regex and OOP logic edge cases before ever hooking up the UI. Furthermore, I used the `with open()` context manager combined with `csv.DictWriter` to safely serialise the complex dictionaries into a flat, readable CSV file, avoiding memory leaks.
* **Lecture 9 (OOP Architecture):** The creation of my domain-specific classes (`UrologyPatient`) deeply expanded on the object-oriented concepts discussed, specifically the usage of `self` for instance attributes, `raise ValueError` for safety mechanisms, and `super()` for hierarchical inheritance.
## 5. Critical Evaluation & Future Enhancements
While the current iteration of Eligibill successfully automates the NICE NG131 screening pipeline, there are clear avenues for future development if the project scope were expanded:
* **Cardiovascular Comorbidity Filtering:** Initially, I planned to design the framework to filter out patients with severe cardiovascular risks (e.g., irregular ECGs or extreme blood pressure), as these often exclude patients from oncology trials. However, I realised that this data is rarely recorded in plain text in standard unstructured narratives (ECGs are waveform data). A future enhancement would involve integrating a separate API to pull structured cardiovascular vitals directly from an EHR to run alongside the NLP text engine.
* **API Integration:** Currently, the system relies on manual batch ingestion of local JSON files. A logical next step would be to build a RESTful API integration to pull live electronic health records (EHR) directly from a mock hospital database.
* **Expanding the NLP Engine:** The deterministic Regex engine is highly accurate for its current scope, but it is rigid. I would look to implement fuzzy string matching (using libraries like `FuzzyWuzzy` or `Levenshtein`) to handle better severe clinical misspellings that might currently bypass the standard Regex catchnets.
* **Database Export:** Instead of exporting to a flat CSV file, the pipeline could be upgraded to utilise `sqlite3` to securely write the audited cohort directly into an encrypted, relational database for better clinical governance.
